# 概要

- `ticket_guy_b_team` は、仕様レビュー駆動のマルチエージェント開発を人間向けに整理して実行するためのフロントエンドである
- 人間は主に CLI (`tgbt`) を通して操作する
- 下層のエージェント実行基盤やワークフローエンジンは内部実装として扱う
- 本アプリケーションの主目的は、エージェントそのものの賢さではなく、仕様化・レビュー・チケット分解・実行統制を扱いやすくすることである

# 基本方針

- 「とにかく自律実行させる」ことよりも、「上流で意図を固定し、下流は自動受け入れゲートで回す」ことを優先する
- 人間は仕様の正しさをレビューし、下流の大量処理は可能な限り機械的に判定する
- 実行基盤の概念をそのまま露出するのではなく、人間目線で理解しやすい業務概念に抽象化する

# 用語

- 仕様書兼実行計画書
    - 人間要望から生成し、レビューと承認を経て後続フェーズの唯一の計画入力になる文書
- チケット
    - 1つの目的と完了条件を持つ実行単位
- 成果物
    - チケットまたは計画の実行結果として生成・更新された、後続工程から参照可能なファイル群
- 自動受け入れゲート
    - テスト、静的検査、成果物存在確認など、機械的に pass / fail を判定できる検証
- エスカレーション
    - 機械判定不能、仕様不足、依存崩れを理由に人間判断へ戻すこと

# 対象範囲

- 本アプリケーションは、仕様策定からチケット実行とレビュー整理までを支援する CLI を提供する
- 実際の LLM 推論やエージェントランタイムの詳細は内部実装として隠蔽する
- 初期バージョンでは単一リポジトリ内での開発支援を対象とし、複数リポジトリ横断実行は対象外とする
- 初期バージョンでは CLI 操作とファイル出力を正とし、Web UI は対象外とする

# 想定ユースケース

- 人間の曖昧な要望から、仕様書兼実行計画書を生成する
- 人間が仕様書兼実行計画書をレビューし、修正を繰り返す
- 承認済みの仕様書兼実行計画書から、実行可能なチケット群を生成する
- チケット群をマルチエージェントで実行する
- 実行結果を、自動受け入れゲートと必要最小限の人間判断で処理する

# ワークフロー

## 第1フェーズ: 仕様書兼実行計画書の草案生成

- 人間の要望を元に、仕様書兼実行計画書の草案を生成する
- 草案は長文になってよい
- この時点では AI エージェントによる実装を開始しない

## 第2フェーズ: 仕様書兼実行計画書のレビューと修正

- 人間による仕様書兼実行計画書のレビューを行う
- この段階のレビューは厳しく行う
- レビュー内容に基づいて AI エージェントは仕様書兼実行計画書を修正する
- 目的は、下流での手戻りやバックトレースを防ぐことである
- 承認されるまで次フェーズに進まず、レビュー・修正を繰り返す

## 第3フェーズ: チケット生成

- 承認済みの仕様書兼実行計画書を元に、AI エージェント用のチケット群を作成する
- チケットには依存関係、成果物、完了条件、検証方法を含める

## 第4フェーズ: チケット実行

- AI エージェント主体で進める
- 人間が逐一レビューすることは前提にしない
- 受け入れ条件・テスト・静的検査・成果物検査を自動受け入れゲートとして使う
- 機械的に判定できない部分だけを人間エスカレーション対象とする

# 状態遷移

- 仕様書兼実行計画書の状態は `draft`、`in_review`、`approved` の 3 状態を持つ
- `plan` で新規生成または更新された仕様書兼実行計画書は `draft` とする
- 人間レビューへ回す段階で `in_review` に遷移させる
- `approved` 済みの計画を再編集した場合は `draft` に戻す
- `approved` でない仕様書兼実行計画書からはチケット生成と実装実行を開始してはならない
- チケットの状態は `todo`、`blocked`、`running`、`review_pending`、`done`、`failed` の 6 状態を持つ
- チケット依存は `ticket_id` だけでなく、依存先に要求する状態も持つ
- 依存条件を満たすまでは `running` に遷移してはならない
- `failed` は自動受け入れゲート失敗または実行中断を示す
- `blocked` は仕様不足、依存未完了、人間判断待ちを示す
- `review_pending` は実行完了後に review ticket または integration ticket の確認待ちであることを示す
- worker ticket は実装とローカル検証完了後に `review_pending` へ遷移し、対応する review ticket が `pass` した後に `done` へ遷移する
- review ticket は対象 worker ticket が `review_pending` になった後に `running` へ遷移できる
- review ticket はレビュー結果ファイルを書き出した時点で完了とみなし、判定結果が `pass` なら `done`、`fail` なら `failed` へ遷移する
- review ticket の判定結果に応じて、対象 worker ticket を `done` または `failed` へ更新する
- review ticket 自体が仕様不足や対象成果物不足で判定不能になった場合は `blocked` に遷移し、その理由をレビュー結果ファイルへ記録する
- integration ticket は依存対象の worker ticket ではなく、依存対象の review ticket が `done` になった後に実行可能とする
- integration ticket は統合対象の成果物をまとめて検証し、統合結果のレビュー結果ファイルを書き出した時点で `done` または `failed` へ遷移する
- root ticket は配下の必須チケットがすべて `done` になった時のみ `done` とする

# 永続化方針

- アプリケーションが扱う主要情報はファイルとして永続化する
- 実装初期段階ではデータベースを必須にしない
- すべての主要オブジェクトは、人間が直接読めるテキスト形式で保存する
- 生成物の既定配置は `artifacts/` 配下とする
- 既定のサブディレクトリは以下とする
    - `artifacts/plans/`
    - `artifacts/tickets/`
    - `artifacts/reviews/`
    - `artifacts/logs/`
    - `artifacts/messages/`

# ファイル仕様

## 仕様書兼実行計画書ファイル

- 1計画につき1ファイルとする
- 保存先は `artifacts/plans/<plan_id>.md` とする
- 先頭メタデータは YAML front matter 形式で記載する
- 先頭に最低限以下のメタデータを持つ
    - `plan_id`
    - `title`
    - `status`
    - `created_at`
    - `updated_at`
- 本文には「仕様書兼実行計画書の要件」で定義した各項目を順番に含める

## チケットファイル

- 1チケットにつき1ファイルとする
- 保存先は `artifacts/tickets/<ticket_id>.md` とする
- 先頭メタデータは YAML front matter 形式で記載する
- 先頭に最低限以下のメタデータを持つ
    - `ticket_id`
    - `ticket_type`
    - `status`
    - `plan_id`
    - `owner_role`
    - `priority`
- `Dependencies` は `ticket_id` と `required_state` の組を列挙する
- 本文には「チケットの要件」で定義した各項目を順番に含める

## レビュー結果ファイル

- review ticket または integration ticket の結果は `artifacts/reviews/<ticket_id>.md` に保存する
- レビュー結果ファイルも YAML front matter と Markdown 本文で構成する
- 最低限以下を含める
    - 対象 ticket_id
    - 判定結果 `pass` / `fail`
    - 確認した受け入れ条件
    - 残リスク
    - 人間エスカレーション要否

## 実行ログ

- チケット実行ログは `artifacts/logs/<ticket_id>-<run_id>.jsonl` に保存する
- 1 回の `ticketguy run` ごとに新しい `run_id` を払い出し、同じ `ticket_id` の再実行でも別ファイルへ保存する
- ログ形式は JSON Lines (JSONL) とする
- 1 行につき 1 イベントを表す JSON object を保存する
- 失敗時だけでなく、成功時も含めて常にログを残す
- 少なくとも以下を記録対象に含める
    - 実行開始時刻と終了時刻
    - 対象 `ticket_id`、`plan_id`、`ticket_type`
    - 実行モード
    - 状態遷移前後の status
    - 実行した外部コマンド
    - 外部コマンドの標準出力
    - 外部コマンドの標準エラー
    - 終了コードまたは判定結果
    - 生成・更新した成果物パス
    - 依存未解決や判定不能などの停止理由
- 各 JSON object には、少なくとも `timestamp`、`event`、`ticket_id` を含める
- 外部コマンド実行イベントでは、少なくとも `command`、`returncode`、`stdout`、`stderr` を含める
- ログは本番環境での事後検証とデバッグに使える粒度で保存し、後から「何を入力に何が起きたか」を追跡できることを要件とする
- ログは人間可読でもあることが望ましいが、一次用途は機械集計・自動解析とする

## 最終メッセージ保存

- `codex exec --output-last-message` の出力は JSONL とは別形式のテキストとして扱う
- この出力は `artifacts/messages/<ticket_id>-<run_id>.txt` に保存する
- JSONL 実行ログと同じファイルへ混在させてはならない

# 仕様書兼実行計画書の要件

- 目的
    - この計画で何を実現したいのかを記載する
    - 解決したい課題、期待する業務上または開発上の効果、完了時の到達状態を含める
- スコープ外
    - 今回は扱わない事項を明示する
    - 類似要求や将来拡張と混同しやすいものを先に除外して、下流での勝手な拡張実装を防ぐ
- 成果物
    - この計画の結果として生成・更新される具体物を列挙する
    - 例として、コード、設定、ドキュメント、テスト、運用手順、生成物配置先などを含める
- 制約
    - 守るべき前提条件や技術的・運用的制約を記載する
    - 使用技術、互換性、性能上限、実行環境、既存仕様との整合、禁止事項を含める
- 受け入れ条件
    - 完了判定に使う観測可能な条件を記載する
    - 人間の主観ではなく、テスト、静的検査、出力確認、状態確認で判定できる形を優先する
- 作業分解
    - 実行時にどの単位へ分けるかを記載する
    - 調査、仕様修正、実装、検証、統合などを分離し、チケット化できる粒度まで落とす
- 未確定事項
    - 現時点で決め切れていない論点を記載する
    - 未確定のまま進めてよい範囲と、確定しないと次へ進めない範囲を区別する
- 依存関係マップ
    - 成果物やチケット間の前後関係を記載する
    - どの作業が何に依存して開始可能になるか、並列化可能な範囲はどこかを示す
- 想定リスク
    - 失敗要因、手戻り要因、ボトルネック候補を記載する
    - 発生条件、影響範囲、事前に打てる軽減策が分かる形にする
- 差し戻し条件
    - どの状態なら上流へ戻すべきかを記載する
    - 仕様不整合、検証不能、未確定事項の顕在化、依存崩れなどの中断条件を明示する
- 検証戦略
    - 何を、どの順序で、何によって検証するかを記載する
    - 単体テスト、結合確認、静的検査、受け入れゲート、人間判断が必要な例外条件を含める

# チケットの要件

- Title
    - チケットの目的が一読で分かる短い名称を書く
    - 対象成果物や作業種別が識別できる名前にし、曖昧な一般名詞だけで終わらせない
- Purpose
    - このチケットが全体計画の中で何を達成するためのものかを書く
    - 実装内容の羅列ではなく、必要性と達成対象を示す
- Inputs
    - 作業開始時点で必要な入力を列挙する
    - 参照仕様、前段成果物、依存チケット、前提設定、既存ファイルを含める
- Outputs
    - このチケットが完了時に生成・更新する成果物を書く
    - ファイル、ドキュメント、検証結果、レポートなど観測可能な形で示す
- Scope
    - このチケットで実施してよい範囲を書く
    - 対象機能、対象ファイル、許可された変更の種類を必要に応じて明記する
- Out of Scope
    - このチケットでは扱わない範囲を書く
    - 似ているが別チケットへ分ける事項を明示して、作業肥大化を防ぐ
- Dependencies
    - 着手または完了判定に先立って必要な依存先を書く
    - 依存チケット、承認済み仕様、外部入力、実行順序制約を含める
    - チケット依存は `ticket_id` と `required_state` をセットで書く
- Steps
    - 実行者が辿る作業手順を書く
    - 調査、編集、検証、成果物保存などを順序付きで示し、途中で確認すべき点も含める
- Acceptance Criteria
    - このチケット単体の完了条件を書く
    - 出力結果、テスト結果、状態変化を用いて、第三者が達成可否を判定できる形にする
- Verification
    - Acceptance Criteria をどう検証するかを書く
    - 実行コマンド、確認対象ファイル、期待結果、必要な人間判断の有無を含める
- Risks / Notes
    - 実行上の注意点や既知リスクを書く
    - 壊れやすい前提、見落としやすい仕様、例外対応方針を簡潔に記載する
- Priority
    - 実行優先度を書く
    - 緊急度ではなく、依存関係や全体計画への影響を踏まえた優先順位として扱う
- Owner または Agent Role
    - 誰が担当する種類の作業かを書く
    - 実装担当、レビュー担当、統合担当など、期待役割が分かる粒度で定義する
- Blocking Conditions
    - 作業を止めてエスカレーションすべき条件を書く
    - 仕様不足、依存未完了、検証不能、権限不足などの停止条件を明示する
- Rollback / Abort
    - 失敗時にどう戻すか、またはどう中断するかを書く
    - 巻き戻し手順、成果物の破棄条件、中断時の報告先や残課題の扱いを含める
- Artifacts Path
    - 成果物やログを保存する場所を書く
    - 実行後に人間や別チケットが参照できるよう、パスや命名規則を具体的に示す

# チケットの分類

## root ticket

- 全体目的
- チケット一覧
- 依存グラフ
- 実行順序ルール
- エスカレーションルール
- 実行対象 plan_id
- 生成する worker / review / integration ticket 一覧

## worker ticket

- 個別タスクの実行専用
- 原則として 1 成果物または 1 変更単位に責務を限定する
- 実装とローカル検証までを担当する

## review ticket

- 成果物検証専用
- 対象チケットの Outputs と Acceptance Criteria の妥当性確認を担当する
- 新規実装は原則として行わない

## integration ticket

- 複数成果物の統合専用
- 複数チケットの成果物を接続し、統合後の検証を担当する
- 各 worker ticket の代替として肥大化してはならない

# チケット運用方針

- 1チケット1成果物を原則とする
- 完了条件は必ず観測可能にする
- 他チケットへの依存関係を明示する
- 「調査」「実装」「レビュー」を1枚に詰め込みすぎない
- 失敗時の差し戻し先を明示する
- 依存先が多すぎるチケットは分割を検討する
- 1枚のチケットで複数ディレクトリにまたがる大規模変更を原則禁止とする
- review ticket は worker ticket と独立して生成できるようにする
- integration ticket は複数 review ticket が `done` になった段階で生成可能とする

# 人間とAIの役割分担

## 人間

- 仕様書兼実行計画書のレビューと承認
- 機械的判定ができない例外の判断
- 最終的な意思決定

## AI

- 草案生成
- チケット分解
- チケット実行
- 自動受け入れゲートの実行
- エスカレーション対象の切り出し

# AI エージェントの行動に関する禁止事項

- 未承認の仕様書兼実行計画書から実装フェーズへ進めてはならない
- 完了条件が曖昧なチケットを実行してはならない
- 自動受け入れゲートなしで大量実行してはならない
- 下流工程を人間の全件レビュー前提で設計してはならない

# CLI 方針

- 人間が触る主要インターフェースは CLI とする
- CLI はフレームワーク固有概念を隠蔽し、このリポジトリ固有の運用概念を提供する
- CLI の起動導線として `bin/ticketguy` を提供する
- 人間は `bin/` を `$PATH` に追加したうえで `ticketguy plan "ここに要件を書く"` のように呼び出す運用を基本とする
- たとえば以下の概念を中心に据える
    - plan
    - approve
    - ticket
    - run
    - review queue
    - artifacts

# CLI コマンド仕様

## `plan`

- 人間要望から仕様書兼実行計画書の草案を生成または更新する
- 入力は自然言語要求、または既存 plan_id と追記指示とする
- 出力は `artifacts/plans/<plan_id>.md` である
- 既存計画を更新した場合は `updated_at` を更新する
- 生成または更新後の状態は `draft` とする
- このコマンドだけでは `approved` へ遷移させない
- 現在の最小実装では、自然言語要求を元に必須 section を並べたテンプレートを生成する
- 現在の最小実装では、要望内容そのものを深く解釈した計画本文生成までは行わず、人間レビュー前提の草案生成に留める

## `approve`

- 指定した plan_id を `in_review` または `approved` へ遷移させる
- 人間レビューへ回す開始操作では `in_review` に遷移させる
- 承認操作では `in_review` の計画だけを `approved` にできる
- 承認前に最低限以下を確認する
    - 必須項目が存在すること
    - 未確定事項が残っていても、次フェーズ進行可否が明示されていること
    - 差し戻し条件と検証戦略が空でないこと
- 不足がある場合は承認せず、理由を表示する

## `ticket`

- `approved` の計画からチケット群を生成する
- 出力は root ticket 1 枚以上と、必要な worker / review / integration ticket 群である
- すべてのチケットに一意な `ticket_id` を付与する
- すべてのチケットに Dependencies と Acceptance Criteria を含める
- review ticket の依存先は対応する worker ticket の `review_pending` とする
- integration ticket の依存先は対象 review ticket の `done` とする
- `approved` でない計画を入力した場合は失敗する
- 現在の最小実装では、1 plan から `worker 1 + review 1 + integration 1 + root 1` の定型構成を生成する
- 現在の最小実装では、plan 本文の作業分解を解析して複数 worker ticket へ自動分割するところまでは未実装である

## `run`

- root ticket または個別 ticket_id を指定して実行する
- 実行対象が root ticket の場合は依存関係に従って実行可能なチケットだけを開始する
- 実行中に `blocked` または `failed` が出た場合は、影響範囲を表示して継続可否を判断できるようにする
- 実行結果として、更新されたチケット状態、ログ、成果物パスを出力する
- 現在の実装では `run <ticket_id> [mode] [model] [reasoning_effort]` の形を取り、`mode` は `dry-run` または `production` を受け付ける
- `dry-run` は外部実行を行わず、最小限の状態遷移と成果物生成だけを確認する
- `production` は worker ticket で `codex exec` を呼び出し、review / integration ticket で自動受け入れゲートを実行する
- 現在の最小実装では root ticket 実行時も並列実行は行わず、依存順の逐次実行に留める
- 現在の最小実装では `blocked` または `failed` の影響範囲分析や継続判断支援の詳細表示は未実装であり、JSON 形式の結果出力に留める
- `run` 実行時は、各段階で起きたことを逐次ログへ出力し、少なくとも開始、依存判定、外部実行、受け入れゲート、状態更新、終了を追跡可能にする

## `review queue`

- `review_pending` のチケット一覧を表示する
- 一覧には最低限以下を含める
    - `ticket_id`
    - `ticket_type`
    - `status`
    - `priority`
    - 依存関係

## `artifacts`

- 指定した plan_id または ticket_id に紐づく成果物パスを表示する
- 存在しない成果物は存在しないことを明示する
- 人間が後続確認しやすいよう、レビュー結果とログも対象に含める

# エラー処理方針

- 仕様不足、承認不足、依存未解決、検証失敗は終了コード非 0 で返す
- エラー表示は、原因、影響範囲、次に取るべき行動を短く示す
- 機械判定不能のため人間判断が必要な場合は、その論点を成果物として保存する
- 失敗時でも、途中まで更新されたチケット状態とログの保存を優先する
- 本番運用でのデバッグ容易性を優先し、正常系・異常系を問わず、実行時に起きた事実をログへ保存する
- ログ出力そのものに失敗した場合も、その失敗を別ログまたは標準エラーへ可能な限り残す

# 現実装との整合メモ

- 仕様上は「チケット群をマルチエージェントで実行する」としているが、現在の実装は単一プロセスの逐次オーケストレーションである
- 仕様上は review ticket と integration ticket の検証戦略を広く取れるようにしているが、現在の最小実装の本番モード受け入れゲートは `pytest -q` を優先し、`tests/` が存在しない場合は成果物存在確認相当の軽量判定に留める
- 仕様上は静的検査も自動受け入れゲート候補に含むが、現在の本番モード `run` は `pyright` 自動実行までは行わない
- 仕様上は review ticket が判定不能時に理由をレビュー結果ファイルへ記録するとしているが、現在の最小実装では対象 worker が `review_pending` でない場合に `blocked` とログ保存を行う実装が中心であり、判定不能理由の粒度は粗い
- 仕様上は integration ticket が複数 review ticket を束ねられる前提だが、現在の最小実装では単一 review ticket を対象にした定型 integration ticket のみを扱う
- 仕様上は plan の自然言語要求から仕様書兼実行計画書を生成するとしているが、現在の最小実装はテンプレート草案生成と人間レビュー前提の補完に依存している
- 仕様上は `run` が root ticket に対して「実行可能なチケットだけを開始する」としているが、現在の最小実装では root 配下の定型チケット列を順番に評価する形であり、一般化されたスケジューラではない
- 仕様上は LangGraph をワークフロー制御に用いるとしており、現在の実装も実際に `langgraph` パッケージの `StateGraph` を使用している
- 仕様上は Codex CLI をバックエンドとするとしており、現在の本番モード worker ticket は `codex exec` を使用する
- 実運用でのデバッグ前提として、現在の仕様では「本番実行時に起きたことを常時ログへ残す」方針を採る
- 本番 smoke test は通常 test suite から切り離し、明示 opt-in のみで実行する
- 本番 smoke test ではトークン消費を抑えるため、現環境で使用可能と確認できた最小クラスのモデルとして `gpt-5.1-codex-mini` を既定候補とする
